# Отчет по учебному проекту
## Сервис рабочих программ (Вариант 13)

### Стек технологий
* **Backend-фреймворк:** FastAPI
* **ORM (База данных):** Peewee (SQLite)
* **ASGI-сервер:** Uvicorn

### Функционал сервиса
* Регистрация и хранение метаданных рабочих программ дисциплин (РПД).
* Привязка программ к внешним специальностям и дисциплинам через транзитивную таблицу назначений (обеспечение связи «многие-ко-многим» в 3НФ).
* Сервис не хранит справочники специальностей и дисциплин, используя только их числовые идентификаторы (интеграция с внешним Specialty Service). Прямая валидация значений `specialty_id` и `discipline_id` на стороне данного сервиса не выполняется.
* Учет версий программ, путей к файлам и поддержка логического (мягкого) удаления.

## На оценку 3: ER-диаграмма и описание сущностей

### ER-диаграмма
База данных спроектирована в третьей нормальной форме (3НФ). Связь типа «многие-ко-многим» между рабочими программами и внешними сущностями реализована через транзитивную таблицу `ProgramAssignment` (на схеме — `ProgramAssignment`), что исключает избыточность данных.

![ER-диаграмма](erd.png)

### Описание сущностей

#### 1. Сущность «Рабочая программа» (WorkProgram)
* **Уникальные комбинации параметров:** `(title, version)` — в системе не может быть двух программ с одинаковым названием и одинаковой версией. Дополнительно уникальным является поле `file_path`.


| Параметр (на английском) | Пояснение | Обязательность | Тип | Ограничение | Значение по умолчанию |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **id** | Идентификатор программы | Да | int | Primary Key, Auto Increment | — |
| **title** | Название программы | Да | string | Not NULL | — |
| **file_path** | Путь к файлу на сервере | Да | string | Not NULL, Unique | — |
| **version** | Версия документа | Да | string | Not NULL, формат x.x | "1.0" |
| **is_deleted** | Флаг мягкого удаления | Да | boolean | Not NULL | False |
| **created_at** | Дата и время создания | Да | datetime | Not NULL | Текущее время сервера |

#### 2. Сущность «Назначение программы» (ProgramAssignment)
* **Уникальные комбинации параметров:** `(work_program_id, specialty_id, discipline_id)` — исключает создание дублирующих связей.


| Параметр (на английском) | Пояснение | Обязательность | Тип | Ограничение | Значение по умолчанию |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **assignment_id** | Идентификатор назначения | Да | int | Primary Key, Auto Increment | — |
| **work_program_id** | ID рабочей программы | Да | int | Foreign Key (WorkProgram.id) | — |
| **specialty_id** | ID внешней специальности | Да | int | Not NULL | — |
| **discipline_id** | ID внешней дисциплины | Да | int | Not NULL | — |
| **is_deleted** | Флаг мягкого удаления | Да | boolean | Not NULL | False |

## На оценку 4: Точки входа REST API (Endpoints)

### 1. Добавить рабочую программу
* **Метод запроса:** `POST`
* **Путь:** `/api/v1/programs`
* **Параметры:** Объект со свойствами `title`, `file_path`, `version`.
* **Назначение:** Создание новой карточки РПД. Возвращает объект программы с присвоенным `id` и `created_at`.

### 2. Получить программу по ID
* **Метод запроса:** `GET`
* **Путь:** `/api/v1/programs/{id}`
* **Назначение:** Поиск информации о конкретной программе.
* **Информация, возвращаемая в случае удачного поиска:** Полный объект программы (если `is_deleted == False`). Если запись логически удалена, возвращается код ответа `404 Not Found`.

### 3. Изменить программу по ID
* **Метод запроса:** `PATCH`
* **Путь:** `/api/v1/programs/{id}`
* **Параметры:** `title` (опционально), `file_path` (опционально), `version` (опционально).
* **Назначение:** Частичное обновление полей рабочей программы (включая возможность изменения версии при её обновлении).

### 4. Удаление программы по ID (Soft Delete)
* **Метод запроса:** `DELETE`
* **Путь:** `/api/v1/programs/{id}`
* **Назначение:** Мягкое удаление программы.
* **Логика работы:** Фактически запись из БД не удаляется, а реализуется через булевое поле `is_deleted`. Полю `is_deleted` присваивается `True`. Для всех связанных назначений в таблице `ProgramAssignment` флаг `is_deleted` также переводится в `True`, чтобы они перестали отображаться в списках.
* **Возвращаемое значение:** Вернет `True`, если статус записи и связанных назначений успешно изменен на «удален», иначе `False`.

### 5. Получить список программ по фильтрам
* **Метод запроса:** `GET`
* **Путь:** `/api/v1/programs`
* **Параметры (Query):** `specialty_id` (int), `discipline_id` (int), `version` (string). Все параметры опциональны.
* **Назначение:** Фильтрация РПД. Выводит только те записи, у которых `is_deleted == False`.

### 6. Удалить назначение программы (Assignment) по ID
* **Метод запроса:** `DELETE`
* **Путь:** `/api/v1/assignments/{id}`
* **Назначение:** Мягкое удаление ошибочной связи. Полю `is_deleted` в таблице `ProgramAssignment` присваивается `True` (запись физически не удаляется).

## На оценку 5: UX приложения (Пользовательский интерфейс)

Взаимодействие с Work Program Service спроектировано для минимизации пользовательских ошибок и обеспечения быстрого отклика системы:

1. **Интерактивная Web-документация (Swagger UI):**
   * Развернута на endpoint-адресе `/docs`.
   * Предоставляет пользователю (разработчику/методисту) визуальные веб-формы для мгновенной отправки запросов, исключая ручной ввод в консоли.
   * Ошибки валидации (например, неверный формат версии) подсвечиваются интерфейсом до отправки запроса в БД.

2. **Сценарий работы пользователя:**
   * **Прозрачность версионирования:** При создании программы пользователю не обязательно думать о версии — система автоматически выставит значение `"1.0"`.
   * **Защита исторических данных (Soft Delete в UX):** Когда методист нажимает кнопку «Удалить программу», она мгновенно исчезает из всех поисковых списков текущего учебного года. Однако благодаря механизму Soft Delete, в архивных отчетах прошлых лет, где фигурировала эта программа, связи и данные не ломаются.
